Project Overview

Modern healthcare systems require efficient coordination across multiple departments, yet many exisiting solutions struggle with fragmented workflows and data silos. The Fullerton Hospital Management System address this challenge by designing a centralized platform streamlines core hospital operations and improves communication between stakeholders.

The system supports a wide range of users, including patients, healthcare providers, administrators, and insurance companies, by integrating essential functionalities such as patient records, appointment scheduling, billing, and inventory management into a unified design.

This project focused on applying software engineering principles to design a scalable and maintainable system using object-oriented design, UML modeling, and GRASP principles. Through an iterative development process, the system was structured to ensure usability, data integrity, and efficient coordination between multiple subsystems in a complex healthcare environment.

Architecture

The system was designed using a layered architecture to separate concerns between the user interface, domain logic, and technical infrastructure. This approach improves maintainability, scalability, and clarity in system responsibilties.

  • UI Layer: Built with React Native for cross-platform compatibility, featuring smooth animations and responsive gesture controls.
  • Domain Layer: Defines the core business logic, including patient management, appointment scheduling, billing, inventory, and health records.
  • Technical Layer: Represents infrastructure-level concerns such as database interactions and external system integration.

This architecture was modeled to support multiple stakeholders and complex workflows while maintaining low coupling between system components.

UML & System Design

UML diagrams were used extensively to model both the structure and behavior of the system before implementation.

  • Domain Model Diagrams: Defined key entities such as PatientManagement, StaffManagement, AppointmentScheduling, Billing, and InventoryManagement, along with their attributes and relationships.
  • Use Case Diagrams & Specifications: Modeled interactions between actors (patients, doctors, admins, insurance) and the system, capturing workflows like patient visits, prescription pickup, billing, and staff scheduling.
  • Sequence & Communication Diagrams: Illustrated how components interact during system operations and how data flows across different layers.
  • System Sequence Diagrams (SSD): Represented high-level system operations between the UI and domain layers.

These models helped validate system behavior and ensure a clear, consistent design before any implementation.

Object-Oriented Design & GRASP Principles

The system design applied key object-oriemted amd GRASP principles to ensure modularity, scalability, and maintainability.

  • Controller Principle: The HealthRecords component was designed as a central controller coordinating interactions across subsystems.
  • Information Expert: Responsibilities were assigned to classes with the necessary data, such as AppointmentScheduling managing appointment creation using patient and staff information.
  • Low Coupling & High Cohesion: Components were structured to minimize dependencies while keeping related functionality grouped logically.
  • Polymorphism: User roles such as doctors, nurses, and administrators were modeled to share common behaviors while supporting role-specific functionality.
  • Indirection: AppointmentScheduling acts as an intermediary between patient and staff components, reducing direct dependencies.

Challenges and Solutions

Challenge: Managing System Complexity Across Multiple Domains

Designing a hospital system required coordinating multiple subsystems (patient management, scheduling, billing, inventory) while ensuring clear responsibilities and interactions. Ensuring that these components worked together without creating tightly coupled dependencies was a major challenge.

Solution: Domain models were structured to isolate functionality and UML diagrams were used to visualize relationships and ensure that each subsystem remained modular and maintainable.

Challenge: Modeling Real-World Workflows Accurately

Healthcare systems involve complex, real-world processes such as patient visits, prescription handling, billing, and insurance claims. Translating these workflows into clear system interactions was difficult.

Solution: Used Use Case Diagrams, Sequence Diagrams, and System Sequence Diagrams (SSDs) to break down workflowws step by step. This helped validate system behavior and ensured that all stakeholder interactions were accurately represented.

Challenge: Balancing Detail vs. Simplicity in UML Diagrams

Creating UML diagrams that were detailed enough to capture system behavior without becoming overly complex or difficult to understand was a key challenge.

Solution: Focused on clarity by abstracting unnecessary details and emphasizing core interactions and relationships.

Lessons Learned

This project reinforced several important lessons about complex software systems design before implementation:

  • Design Before Implementation Matters: Working through UML diagrams and system modeling highlighted the importance of planning system architecture before writing code. A well-structured design makes implementation more predictable, scalable, and easier to maintain.
  • Simplicity Improves Design Quality: Overly complex UML diagrams can reduce clarity and usefulness. Learning to balance detail with readability helped improve both communication and overall system design.
  • Modeling Real-World Systems: Translating real-world healthcare workflows into software design required breaking down complex processes into manageable components. This emphasized the importance of abstraction and iterative refinement when dealing with large systems.
